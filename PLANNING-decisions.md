### 2026-06-05 — Source grounding: keep Perplexity snippets (full-text crawling deferred), block video/social hosts, make the eval source-aware
**Context**: The first live v2 Deep Dive run (manual, on the `automated-work` sandbox) produced a competent post that nonetheless asserted specific figures ("20+ hours/month", "30% of queries") not clearly traceable to its sources. That prompted an investigation into *how* the pipeline grounds posts at all. Key facts established:
- **Claude never fetches a URL.** It has no web access in this pipeline. Every post is synthesized entirely from what Perplexity Sonar `/search` returns per result: `title`, `url`, `date`, and a `snippet`. The blog writer serializes those Article objects straight into the prompt (`a.model_dump`).
- **We were already at Perplexity's max extraction depth.** `search_context_size` (`low`/`medium`/`high`) is the real content-depth lever; `high` is the default and we don't override it. Empirically (one Pricing & Analytics query): low ≈ 228 total words, medium ≈ 784, high ≈ 3,077 — high returns ~400–600 words per article, which is the **front section** of each piece. `max_tokens_per_page` is a *downward cap only*: setting it to 4,000 / 8,000 produced byte-identical output, so it cannot pull *more* content.
- **A/B on full text.** Took 5 real article URLs from a live search, fetched their full bodies (via WebFetch as a crawler proxy), and generated a Deep Dive two ways from the *same* sources — snippets vs full text. Full-text output was richer and more precise (it surfaced figures the snippet truncated: CarGurus QARSD $7,500/qtr, 85% inventory share, 68.4% ChatGPT reliance), but the lift was **modest and article-dependent**, not transformative. These trade articles front-load their stats, so `high` snippets already capture most key numbers; full text mattered most for long/deep pieces whose snippet was truncated (e.g. the Cars.com earnings transcript, snippet only 204w).
**Decision**: Three outcomes from the investigation.
1. **Keep Perplexity `high` snippets as the grounding source; defer building a full-text crawler.** No code change — we confirmed we're already extracting maximally from `/search`. If grounding quality later demands more, the chosen fallback is a **selective crawl**: fetch the full page *only* when a snippet comes back short (≈ <150 words), capturing most of the benefit at a fraction of a full crawler's cost/complexity.
2. **Block video/social hosts in news sourcing** (committed `c56b786`). YouTube et al. are structurally thin source material — a video's snippet is a description/transcript fragment, not reported text — so they're dropped at the host level before the classifier. `_is_blocklisted` now suffix-matches subdomains (`m.youtube.com`) without catching lookalikes (`notyoutube.com`); this also extends the existing PR-wire entries to their subdomains.
3. **Make the generation eval verify against source excerpts** (committed `f545779`). The eval judge had been given only `title`/`publisher`/`url`, never the source text — so it scored *every* specific number as unverifiable and failed well-grounded posts, calling real, in-source facts "invented." Before/after on the same posts: blind → `grounding=0/fail` ("no excerpt available — zero claims can be verified"); source-aware → `grounding=2/pass`. The judge prompt now renders an `EXCERPT` per source; sources without one render a placeholder rather than counting as proof of invention.
**Rationale**:
- **A crawler isn't justified by the evidence.** The quality lift is modest for our mostly front-loaded trade sources, while a crawler is expensive and brittle (HTML parsing, paywalls, JS-rendered pages, robots.txt, reliability, maintenance, more tokens/run). Tell: WebFetch itself had to *summarize* the long Cars.com transcript rather than return it verbatim — even "just crawl it" isn't clean on long pages.
- **Blocklist over relying on the classifier alone for video.** A video is thin regardless of who posted it; a deterministic host-level drop is cheaper and more reliable than asking the Haiku classifier to judge it, and it shrinks the classifier's input.
- **A source-blind eval can't measure the one axis it exists for.** Grounding is the whole point of the eval; without source text it was structurally unable to score it and defaulted everything to fail. Feeding excerpts is the minimum that makes the grounding axis meaningful — a bigger, cheaper win than chasing deeper sources.
**Tradeoffs**:
- **Grounding ceiling is now explicit and accepted.** Posts are built from the top ~third of each article. A fact buried deep in a long piece genuinely isn't available to the model — that's the residual risk the selective-crawl fallback exists to address if it ever bites.
- **Eval fix is capable but not yet fed in production.** `evaluate_post` can now use excerpts, but its only caller (`eval_recent_posts.py`) reads *persisted* posts, and the `sources` table doesn't store the excerpt — so in production it still renders "(no excerpt available)". Closing this needs either **in-loop evaluation** right after generation (Articles still in hand — gate-free) or a **new `sources.excerpt` column** (schema change → architecture gate). Deferred pending a decision; the eval stays a runnable QA tool, not a wired publish gate.
- **Video blocklist is a small static list.** It enumerates the obvious offenders (YouTube, Vimeo, TikTok, Reddit, X/Twitter, Facebook, Instagram); a new video host slips through until added. The classifier remains the backstop behind it.
- **Snippet depth is Perplexity's call, not ours.** If Sonar changes what `high` returns, our grounding silently shifts. The volume probe (which records raw/news counts weekly) would surface a gross change but not a subtle depth regression.

### 2026-05-29 — Backend auth: shared-secret gate + proxy session check (not JWT verification)
**Context**: A QA audit on 2026-05-29 found the backend enforced **no auth on any endpoint** — `require_api_key` did not exist, and the only auth check anywhere was `frontend/src/proxy.ts` guarding the `/dashboard/*` *pages*. The FastAPI service (public on Render) would execute any admin/pipeline/settings write from an unauthenticated request — e.g. `curl -X POST $BACKEND/pipeline/run` spends real Claude + Perplexity money. CORS does not mitigate this (it only constrains browser JS, not `curl`/server-side callers). The `backend-auth` feature closes the gap.
**Decision**: Two enforcement points, one shared secret (`BACKEND_API_SECRET`):
1. **Backend** — a `require_api_key` FastAPI dependency (`backend/dependencies.py`) applied at router level to `posts`, `pipeline`, and `settings` via `APIRouter(..., dependencies=[Depends(require_api_key)])`. It requires `Authorization: Bearer <BACKEND_API_SECRET>`, compares constant-time (`hmac.compare_digest`), returns 401 on missing/wrong, and 500 if the secret env var is unset (fail closed). `/public/*`, `/auth/*`, and `/health` stay open.
2. **Frontend proxy** — a shared `proxyToBackend` helper (`frontend/src/lib/proxy-backend.ts`) used by every `/api/*` route that targets a protected endpoint. It verifies the NextAuth session via `getToken` (401 if logged out), then attaches the Bearer secret on the server-to-server fetch. The secret is server-env only (root `.env`, `frontend/.env.local`), never sent to the browser.
**Rationale**:
- **Shared secret over verifying the NextAuth JWT at the backend.** For a single-admin app the backend doesn't need per-user identity — it only needs to know the request came from our trusted frontend. A static shared secret is a few lines and no new deps. Verifying the NextAuth JWT would couple the Python backend to NextAuth's token format/signing and the `NEXTAUTH_SECRET`, for identity we don't use.
- **Defense in depth.** The backend secret is the real trust boundary (stops direct `curl` against Render). The proxy session check stops the `/api/*` routes being an open relay for a logged-out browser. Either alone leaves a hole; both together close it.
- **Router-level dependency** so the gate can't be forgotten on a new route added to a protected router — it applies to every path in the router by construction.
- **Fail closed on missing secret** so a misconfigured deploy returns 500, never silently serves protected routes unauthenticated.
**Tradeoffs**:
- **One static secret, no rotation/expiry.** Rotating means updating Render + Vercel together. Acceptable at single-admin MVP scale; a rotation/KMS story is explicitly out of scope (see the feature plan).
- **No rate limiting / brute-force protection** on the secret. A 32-byte random secret is infeasible to guess, but there's no lockout. Noted as possible future hardening.
- **Prod rollout coupling.** The secret must be set identically on Render (backend) and Vercel (frontend) or every admin action 500s/401s. Flagged as a dependency that rides with the `deploy` work.
- **No per-user audit trail.** The backend learns "trusted frontend," not "which admin." Fine for one operator; revisit if multi-user ever lands (it's currently out of scope project-wide).

### 2026-05-24 — News-fetcher quality filter: per-tag queries, domain allowlist, per-cluster threshold
**Context**: First real smoke test of `news-fetcher` on 2026-05-15 returned 10 articles, 3 of which were vendor product pages (`owini.ai`, `drivecentric.com`, `spyne.ai`) rather than news. Generated posts were also blending 3–4 categories per post, diluting topical authority. The `news-fetcher-quality-filter` feature (Phase 5) covered both problems: surface only real news *and* keep posts focused on one topic.
**Decision**: Three-layer rewrite of `backend/services/news_fetcher.py`:
1. **Query phrasing** — replaced the 5 original keyword-style queries with 7 `news:`-prefixed queries, each mapped 1:1 to one of the 7 official post tags (Voice AI, CRM, Merchandising, Industry Move, Pricing & Analytics, Sales Dev, OT & Infrastructure). Dropped the original broad "AI tools being used in car dealerships" catch-all — it didn't fit the per-cluster model and its conceptual coverage is already split across the 7 specific queries.
2. **Domain allowlist** — Sonar `search_domain_filter` set to ~18 curated automotive trade press, dealer-focused publications, general tech, and business news outlets. Replaces a hypothetical blocklist as the second-layer safety against vendor pages.
3. **Per-cluster threshold** — switched Sonar to one HTTP call per query (not batched) so each result's source query and tag is known by attribution rather than re-classification. Articles deduped by URL (first-seen wins), grouped by tag, largest cluster wins; ties broken by preferring a tag different from the most recently published post. The ≥3 article threshold applies to the **winning cluster**, not the raw pool — a run where no single category reaches 3 articles is skipped.

Also narrowed the blog-writer prompt from "select 2–4 tags" to "select 1–2 tags," and added a sentence explaining the input is pre-filtered to one category so the model defaults to one tag.
**Rationale**:
- **Tighter queries first.** `news:` is a known Sonar signal that pushes ranking toward editorial results. Combined with year refs (`2026`) and industry-specific jargon (`BDC`, `DMS`, `pricing intelligence`), the query phrasing alone removes most vendor-page noise before any post-filter runs.
- **Allowlist beats blocklist.** A blocklist has to enumerate every vendor that might ever appear (unbounded). An allowlist of ~18 known news outlets is bounded and gets stronger over time as we add outlets the operator trusts. Sonar caps `search_domain_filter` at 20 entries, which is plenty.
- **Per-query HTTP calls over batched.** Sonar's batched mode (`query: string[]`) doesn't tag results with their source query in the response. Per-query attribution would otherwise require classifying every snippet after the fact (a Claude call per article, or fragile heuristics). 7 sequential HTTP calls runs twice a week — negligible cost, clean attribution.
- **Per-cluster threshold over pool threshold.** A pool of 8 articles spread across 4 categories (2/2/2/2) is *not* the same as 3 articles in one category — the first produces a fragmented post mixing 4 topics, the second produces a focused post about one. Counting per cluster enforces the editorial principle that "≥3 articles" means "we can credibly identify a pattern in *one topic area*" — same logic as the original threshold rationale (2026-05-13 decision), now applied at the right granularity.
- **3-lite over 3-full.** When multiple clusters tie or all hit ≥3, we generate only **one** post (from the winning cluster) and discard the rest of the fetched articles. The richer alternative — generate posts for all qualifying clusters and route them through the queue — was deliberately deferred to a post-MVP feature (`multi-cluster-pipeline`) because it cracks open `publishing_mode = auto` semantics (immediate-publish vs scheduled-slot-fill) that need a proper design pass. For MVP, wasted Sonar fetches on losing clusters are cheap; staying single-post-per-run keeps the existing pipeline, scheduler, and admin queue contracts intact.
- **1–2 tag cap on blog-writer.** With pre-filtered single-category input, 3+ tags is now a sign the model is overreaching. Cap at 2 to allow a sensible secondary tag (e.g. a CRM article that also genuinely touches Sales Dev) without letting tag count drift back up.
**Tradeoffs**:
- **8th query dropped.** The original "8 queries total" approved during planning collapsed to 7 once the broad catch-all stopped fitting the per-cluster model. Tag coverage is unchanged (the 3 new queries already cover the formerly-uncovered tags). If the broad query later proves useful for surfacing emerging trends outside the 7 categories, it can come back as a separate "exploratory" channel that doesn't compete for cluster selection.
- **Tie-break uses primary tag of last published only.** If the last post had two tags, the tie-break only considers `tags[0]`. Simple, deterministic, and the primary tag is typically the focused-cluster tag anyway. If this gets wrong-headed in practice (e.g. consistently rotating only between two tags), revisit.
- **First-seen URL dedup is order-dependent.** When the same article URL is returned by two queries, the earlier query (per `QUERY_TO_TAG` iteration order) wins the tag assignment. The dict is defined in a deliberate priority order; if the ordering ever gets contentious, switch to "tag whichever query's result has the higher rank/score" once Sonar exposes a stable ranking signal.
- **Allowlist may be too tight in week one.** ~18 outlets is a conservative starting point. If real runs frequently come up empty or below threshold, the allowlist is the first knob to loosen (add more outlets) before relaxing the threshold itself.

### 2026-05-17 — Scheduled-post auto-publisher: 1-minute polling, atomic UPDATE
**Context**: `scheduled-and-published` (Phase 3) shipped the UI for "schedule for later" but no background worker to actually flip a post from `accepted` to `published` when its `scheduled_at` arrives. The gap surfaced during manual testing on 2026-05-17 when a post scheduled for 7:50 PM was still sitting on `/dashboard/scheduled` four minutes later. The roadmap had no explicit feature for this — an oversight in product planning, not something that existed and broke. This is the follow-up `scheduled-publisher` feature that closes it.
**Decision**: Add a second job to the existing `AsyncIOScheduler` in `backend/scheduler.py` using `IntervalTrigger(minutes=1)`. The job calls a new `publish_due_posts(db)` service function in `backend/services/publisher.py`. That function does a single atomic SQLAlchemy `update()` filtering on `status='accepted' AND scheduled_at <= now()`, setting `status='published'`, `published_at=now()`, and clearing `scheduled_at`. No new tables, no new columns, no new packages, no new env vars. The job logs only when at least one post was published (silent on empty scans); errors are logged via `logger.exception` and swallowed so the scheduler thread can never die.
**Rationale**:
- **Polling beats per-post one-off jobs.** APScheduler can schedule a one-off job *at* each post's `scheduled_at`, but that puts scheduler state in two places (DB + APScheduler jobstore) that have to be reconciled on every restart, edit, and back-to-queue. With polling, the DB is the only source of truth and the worker is stateless. On Render restart, the next tick picks up any posts whose time passed during downtime — self-healing without ceremony.
- **1-minute cadence is a UX choice, not a cost concern.** Each tick is one indexed `UPDATE` returning 0–1 rows. No LLM calls (publishing only flips status — content was generated when the post first entered the queue). Render backend is already always-on for the Mon/Thu cron; adding a second job doesn't change the bill. Neon free tier is metered on compute time, not query count, and 1,440 sub-10ms queries per day is rounding error. The 60-second worst-case lag is acceptable for a twice-weekly publication.
- **Atomic UPDATE filtering on `status='accepted'` in WHERE is the race-safety mechanism.** A concurrent manual `POST /posts/{id}/publish` and the worker can both target the same row, but whichever transaction commits first wins; the loser's UPDATE matches 0 rows. No row-level lock needed, no 409s leaked from inside the worker, no double-publish. This is the bulk-UPDATE equivalent of the single-row 409 status guard already used by `/accept`, `/reject`, etc. in `routers/posts.py`.
- **Quiet logging** because 1,440 ticks/day at INFO level would drown real signal. Only logging on `count > 0` (and on exceptions at ERROR) gives us a clean "scheduled-publisher published N post(s)" trail of actual work, nothing else.
**Tradeoffs**:
- **No retry machinery / no per-post error column.** If a single UPDATE fails mid-tick, the next tick (60s later) retries automatically because the same WHERE filter still matches. If failures are persistent (e.g. DB down for hours), the post just stays `accepted` and the issue surfaces in operator monitoring — not a per-post UI alert. Acceptable for MVP; can layer on later if needed.
- **Single-worker assumption inherited from the existing Mon/Thu cron.** Both jobs assume one backend process owns the scheduler. Already a CLAUDE.md gotcha; adding a second job doesn't change the constraint, just inherits it. Production is single-instance on Render Starter, so not a current problem.
- **Misfire grace = 300s.** If the event loop is blocked or the box restarts, a missed tick within 5 min still fires once (coalesced); past that it's skipped. Combined with the "scan all due posts" design (not just "due in this minute"), the scheduler can skip several ticks without losing posts — the next successful tick catches everything overdue.
- **No admin-UI surface for publish failures.** Server logs only. If the operator wants visibility into "did the worker publish what I scheduled?" they check `/dashboard/published` — the visible state IS the success signal. Failed publishes are silent except in logs.

### 2026-05-17 — Phase 3 reordered: overview lands last, not first
**Context**: The original Phase 3 sequence opened with `dashboard-shell-and-overview` — bundling the shell (sidebar + main layout) with the `/dashboard` overview page. While auditing the feature plan we discovered the overview's 4 stat cards summarize data (`posts` counts, publishing mode) that lives in tables exposed by *later* phase-3 features. Building overview first would force this feature to either (a) build `GET /posts` and `GET /settings` endpoints that logically belong to `review-queue` and `settings-page`, or (b) ship stat cards with placeholder data that don't meet the success criterion "values pulled from the backend."
**Decision**: Split the feature into `dashboard-shell` (shell + tokens + proxy + chamfer primitive + placeholders) and `dashboard-overview` (stat cards + quick actions). Reorder Phase 3 to: `dashboard-shell` → `review-queue` → `scheduled-and-published` → `settings-page` → `dashboard-overview`. Each feature now builds the endpoint it directly needs; the overview lands at the end as a thin composition layer over endpoints that already exist.
**Rationale**:
- **Endpoint emergence is natural, not forced.** `review-queue` needs the *list* of pending posts (so `GET /posts` is built there, with a `total` field that doubles as the count source overview later consumes). `settings-page` needs `GET /settings` to render the publishing-mode toggle. Each endpoint is built by the feature that's its primary consumer.
- **No interim functionality loss.** `Design/README.md` already puts a `TRIGGER MANUAL RUN` button on the Settings page — the operator gets a way to fire pipelines once `settings-page` ships, well before overview. Every phase-3 page is independently usable via the sidebar.
- **Smaller, more reviewable shell PR.** Shell becomes pure UI + foundation (tokens, fonts, body grid, `ChamferedPanel` primitive, `proxy.ts`, placeholder pages). No data fetching except `GET /pipeline/status` (already built in Phase 2) for the bottom status dot.
- **Sidebar QUEUE badge** ships as a hardcoded `0` slot in the shell feature and gets wired live during `review-queue` — the chrome stays stable, only the data source changes.
**Tradeoffs**: The operator no longer gets a glanceable cockpit *first* — the summary view is the final phase-3 deliverable. Acceptable because: (1) each intermediate page does its own job standalone, and (2) building overview first would have meant either fake data or premature endpoints, both worse than waiting. Out-of-scope blockers also folded into the shell feature: design tokens were stale (still on the pre-v2.0 blue palette) and `/dashboard*` had no auth gate — both are small and trivially included in the shell PR rather than spun out as additional follow-ups.

### 2026-05-13 — News fetcher minimum article threshold: ≥3 qualifying articles per run
**Context**: `news-fetcher` (Phase 2) needs a floor on how many articles must survive Sonar's response (after URL-dedup) before a run proceeds to `blog-writer`. The original spec mentioned "≥3" without explaining why. Operator asked for the rationale before shipping, so the value lives in code as `MIN_ARTICLES = 3` with the reasoning captured here rather than inline.
**Decision**: A pipeline run requires **≥3 unique articles** (after URL-dedup) to proceed. Below the threshold, `fetch_qualifying_articles()` returns an empty list and the orchestrator (when it lands) skips the run, logged but not retried. The threshold is a module-level constant in `backend/services/news_fetcher.py` — not exposed in the Settings UI, not stored in the `settings` table. Editorial decisions like this stay in code, consistent with how the twice-weekly cadence is handled.
**Rationale**:
- **1 article** → the post would be a paraphrase of one news item. No synthesis, just a rehash. Below the editorial bar.
- **2 articles** → can compare/contrast, but fragile. If both articles cover the same incident from different angles, you effectively have 1 story.
- **3+ articles** → can credibly identify a *pattern* or *trend*. Three independent data points start to suggest a theme. The post can say "here's what's happening across the industry" instead of "here's one thing that happened."

The blog's audience (dealership operators, automotive tech executives) wants analysis, not aggregation. Sources transparency is part of the editorial contract — a list of 3+ sources signals "we did our homework." Two would still publish credibly, but the post would be thinner.
**Tradeoffs**: Three is a defensible default, not a hard truth — you could ship at 2 or 4. Hardcoding it (rather than putting it in `settings`) means an editorial taste change is a one-line PR, not a UI toggle. Worth revisiting after a few months of real runs: if Sonar consistently returns 6–10 qualifying articles per run, the threshold isn't the binding constraint and can stay where it is; if it's bumping up against 3 frequently, that's a signal to either widen the intent queries or accept thinner posts (lower the threshold to 2).

### 2026-05-13 — News fetcher uses Perplexity `/search`, not `/chat/completions`
**Context**: `news-fetcher` (Phase 2) is starting implementation. Perplexity exposes two relevant endpoints — the OpenAI-compatible `POST /chat/completions` with `model: "sonar"` (returns a synthesized prose answer plus citations), and a newer dedicated `POST /search` (returns raw retrieved articles, no synthesis). The original "Pipeline logic / Step 1 — News fetching" section said "send intent-based queries to Perplexity Sonar" without specifying which endpoint, leaving the choice open.
**Decision**: Use `POST https://api.perplexity.ai/search`. Send all 5 intent queries in a single request via the endpoint's `query: string[]` parameter (batched). Apply `search_recency_filter: "week"` (closest match to the "past 2 weeks" intent the queries imply). Parse the `results[]` array into a Pydantic `Article` model (`title`, `url`, `publisher`, `published_date`, `snippet`). Derive `publisher` client-side from the URL hostname (`urlparse(url).hostname`, strip leading `www.`) since neither Sonar endpoint returns it as a field. Hostname → friendly-name mapping (e.g. `techcrunch.com` → "TechCrunch") is deferred — ship with raw hostnames and revisit only if source citations look ugly in real posts.
**Rationale**: Perplexity's semantic + LLM-ranked retrieval ("hybrid search ... LLM ranking ... documents divided into fine-grained units individually surfaced and scored") happens in the retrieval layer that sits **beneath** both endpoints. `/chat/completions` adds an LLM synthesis step **on top** that produces a prose answer — content the `blog-writer` step will regenerate from scratch with Claude anyway, so we'd discard it. `/search` skips that wasted synthesis step, returns articles directly as a structured array, and accepts batched queries (one HTTP call instead of five — lower latency, fewer failure modes). The original "Sonar replaces a build-our-own news fetcher + relevance filter" rationale is preserved because the relevance filtering capability lives in the retrieval layer, not the synthesis layer. Cost difference is a rounding error (fractions of a cent per run) — not a decision factor.
**Tradeoffs**: If Perplexity ever differentiates ranking quality between the two endpoints (e.g. gives `/chat/completions` exclusive access to a higher-quality reranker), `/search` becomes the lesser product. Not currently documented but possible. Mitigation: the client wrapper isolates the endpoint choice in one module so swapping is cheap. Neither endpoint returns `publisher` — deriving from hostname gives `techcrunch.com` not `"TechCrunch"`; readers see hostnames in source citations until/unless we add a mapping table. Recency filter `"week"` doesn't perfectly match the original "past 2 weeks" phrasing in the queries — the next-best option is `"month"`, which is too wide; we accept the slight narrowing.
**References**:
- API reference: https://docs.perplexity.ai/api-reference/search-post
- Sonar quickstart: https://docs.perplexity.ai/docs/sonar/quickstart
- Search API announcement: https://www.perplexity.ai/hub/blog/introducing-the-perplexity-search-api

### 2026-05-09 — Cadence change: twice-weekly (Mon + Thu) replaces bi-weekly; both days and frequency are hardcoded
**Context**: Original spec called for a bi-weekly cadence (every other Monday). Operator clarified the actual goal is two posts per week. Forcing the conversation early — before `scheduler-cron` is built — keeps the change cheap (only docs + the seeded settings row exist; no scheduler code yet).
**Decision**:
- Pipeline fires **every Monday and every Thursday at 8:00 AM**. Both days and the frequency are constants in the scheduler code, not settings.
- The `settings.schedule_day` column is dropped (it had no purpose once days became fixed). `settings.schedule_frequency` is kept as a VARCHAR but its value is updated from `'biweekly'` to `'twice_weekly'` — vestigial today, but cheaper to retain than to rip out the column and update every reference.
- The Settings UI no longer exposes a day-of-week selector. Schedule is rendered read-only as `MON + THU AT 8:00 AM`.
- Approximate run volume: ~8–9 runs/month (~104/year). API costs (Perplexity Sonar + Claude Sonnet 4) remain well under $5/month at this volume; hosting still dominates total cost.
**Rationale**: Two fixed days is the simplest possible scheduler — a single APScheduler `CronTrigger(day_of_week='mon,thu', hour=8)` handles it. Removing both knobs from the UI is consistent with the existing "frequency is fixed, not configurable" philosophy and reduces the settings-page scope. Doing this *before* the scheduler is built avoids writing throwaway day-of-week PATCH plumbing.
**Tradeoffs**: If a future operator wants Tue + Fri (or any other pair), it's a code change — not a settings change. That's acceptable at single-operator MVP scale; revisit if/when a second operator joins or operator preferences shift. The retained `schedule_frequency` column is technically cruft (only one valid value), but dropping it would force a settings API change for no user-facing benefit.

### 2026-05-09 — Auth: NextAuth owns the session; FastAPI is stateless; Next 16 `proxy.ts` enforces the gate
**Context**: `auth-login` feature ships the auth boundary. Two design choices needed pinning before downstream protected endpoints land.
**Decision**:
- **Session ownership**: NextAuth (credentials provider, JWT strategy, 2h absolute `maxAge`) owns the session. FastAPI's `POST /auth/login` only verifies bcrypt against the `users.hashed_password` column and returns the user JSON; it does **not** issue a session cookie. NextAuth's `authorize()` callback runs server-side inside the Next.js process and POSTs to FastAPI from there — no CORS plumbing needed for this feature. The shared `NEXTAUTH_SECRET` lives in `frontend/.env.local`.
- **Auth gate**: Next.js 16 renames `middleware.ts` → `proxy.ts` (functionality unchanged, runs on Node.js runtime). The project uses `frontend/src/proxy.ts` with `getToken` from `next-auth/jwt` directly, instead of `next-auth/middleware`'s `withAuth` wrapper — more portable and version-independent. Matcher: `/dashboard/:path*`.
- **Future protected backend endpoints**: when a feature first needs a backend route that's only callable by the admin (e.g. `POST /pipeline/run`), the simplest option is for FastAPI to verify the NextAuth JWT directly using the shared `NEXTAUTH_SECRET` — no separate FastAPI session system. To revisit when that feature lands.
- **Out of scope**: sliding/idle session refresh. 2h is a hard absolute cap. Sliding-2h-idle is a future option (~5-line change in NextAuth JWT callback) if real usage hurts.
**Rationale**: Single source of truth for sessions (NextAuth) avoids two competing session systems. Using `getToken` directly in `proxy.ts` keeps the auth gate independent of next-auth's `next/server` integration version. Deferring the FastAPI side of JWT verification keeps this feature small while leaving a clear next step.
**Tradeoffs**: Future devs reading the FastAPI router will see `/auth/login` but no session middleware on protected routes — they need to know the auth check happens upstream in the Next.js layer (or, eventually, in a FastAPI dependency that decodes the NextAuth JWT). The 2h absolute cap may surprise admins mid-task; sliding is documented as a deliberate future option, not an oversight.

### 2026-05-09 — Frontend stack: Next.js 16 + React 19 + Tailwind v4
**Context**: `frontend-skeleton` feature scaffolds the Next.js app. The original spec called for Next.js 14, but `create-next-app@latest` now installs Next 16 + React 19 + Tailwind v4. No code yet depends on Next 14 specifics.
**Decision**: Adopt the current stable stack — Next.js 16 (App Router) + React 19 + Tailwind v4 — instead of pinning to Next 14. Source layout is `frontend/src/app/`. shadcn/ui initialized at scaffold time (`components.json` + `lib/utils.ts` only); no components installed yet — features that need them install on demand. Design tokens live in `frontend/src/app/globals.css` as CSS custom properties and are surfaced as Tailwind utility classes via the `@theme {}` directive (Tailwind v4's CSS-first config). There is no `tailwind.config.ts`. Fonts (Inter + JetBrains Mono) are loaded via `next/font/google` and exposed as `--font-inter` / `--font-jetbrains-mono`, then mapped to Tailwind's `font-sans` / `font-mono` inside `@theme`.
**Rationale**: No reason to pin to an older major when nothing depends on it yet; future features build on the latest stable APIs. Tailwind v4's CSS-first theme matches the design-token-driven approach in `Design/README.md` cleanly — tokens stay in one file, in CSS.
**Tradeoffs**: Next 16 has breaking changes from Next 14/15 (see `frontend/AGENTS.md`) — anyone with Next 14-era muscle memory should consult the bundled docs in `frontend/node_modules/next/dist/docs/` before writing new code. The Tailwind v4 `@theme {}` pattern is unfamiliar to anyone who only knows v3's `tailwind.config.ts`. shadcn components (when added later) will reference shadcn's expected token names (`--background`, `--foreground`, etc.) — the first feature that adds a component will need to alias those to our design tokens.

### 2026-05-09 — Database foundation: native ENUMs, app-side UUIDs, sync ORM, settings seeded in migration
**Context**: `database-foundation` feature ships the persistent data layer (4 SQLAlchemy models + Alembic baseline + `seed_admin.py`). Several small choices needed pinning before downstream features depend on them.
**Decision**:
- Use Postgres native `ENUM` types for `post_status` and `publishing_mode`, defined once and reused across `posts.status`, `posts.publishing_mode`, and `settings.publishing_mode`. The initial migration creates each type explicitly via `postgresql.ENUM(..., create_type=False).create(bind, checkfirst=True)` so columns can reference them without duplicate `CREATE TYPE` errors; `downgrade()` drops the types after the tables.
- Generate UUID primary keys app-side via `uuid.uuid4` as the SQLAlchemy column default — no `pgcrypto` / `uuid-ossp` extension required.
- Seed the single `settings` row directly in the initial migration. Means `alembic upgrade head` is sufficient; no separate seed-settings step. (The original seed used `schedule_frequency='biweekly'` and a `schedule_day='monday'` column; both were superseded on 2026-05-09 when cadence switched to twice-weekly — see decision log entry of that date.)
- Use sync SQLAlchemy 2.x + `psycopg2-binary`. `database.py` exposes `engine`, `SessionLocal`, `Base`, and a `get_db()` generator for FastAPI's `Depends`.
- `passlib[bcrypt]==1.7.4` with `bcrypt==4.0.1` pinned. The 4.0.1 pin avoids the noisy `(trapped) error reading bcrypt version` warning that appears with passlib 1.7.4 + bcrypt ≥ 4.1.
**Rationale**: Native ENUMs give DB-level integrity — invalid values can't slip in via raw SQL. App-side UUIDs avoid extension management and work identically across environments. Seeding `settings` in the migration makes the post-migration state self-sufficient (no "remember to also run X"). Sync ORM matches the rest of the FastAPI app's sync style and keeps the learning surface smaller for now.
**Tradeoffs**: Adding a value to `post_status` or `publishing_mode` later requires an explicit `ALTER TYPE ... ADD VALUE` migration — not as flexible as VARCHAR + CHECK. Sync sessions mean blocking I/O in request handlers; if request volume grows, swapping to async SQLAlchemy is a discrete future migration. The bcrypt 4.0.1 pin is a known-good silence-the-warning workaround; revisit when passlib publishes a release that handles bcrypt 4.1+ cleanly.

### 2026-05-08 — Postgres 17 pinned; host port 5433 in local Docker
**Context**: `backend-skeleton` feature scaffolds `docker-compose.yml`. Developer machine already runs Homebrew Postgres 17.7 as a launchd service bound to host port 5432.
**Decision**: Pin the `db` service to `postgres:17-alpine` and map host port **5433 → container 5432**. Container-internal connection string remains `postgresql://...@db:5432/...`; host-side tools connect via `localhost:5433`.
**Rationale**: Matching major versions (host client 17.7 ↔ container server 17.x) avoids client/server warnings and behavior drift. Host port 5433 sidesteps the bind conflict without requiring the developer to stop the local Postgres service.
**Tradeoffs**: Anyone joining the project who *doesn't* have a local Postgres on 5432 will still connect via 5433 — slight surprise, but documented in `.env.example` and PLANNING.md. If the project later moves to managed Postgres on Railway/Render, the host port mapping is irrelevant.
