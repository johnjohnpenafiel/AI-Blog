### What shipped
- `backend/taxonomy.py` — single source of truth for the v2 vocabulary: `SECTION_TAGS` (9 sections → nested tags), derived `SECTIONS`/`TAGS`, `FORMATS` (6), `STORY_TYPES` (3), plus `is_valid_*` helpers and `tags_for_section()`.
- `posts` gains three nullable text columns — `section`, `format`, `story_type` — with `@validates` guards on the model that reject out-of-vocabulary values on assignment (and don't fire on DB load).
- `PostOut` + `PostListItem` now expose the three fields (read side); new `PostTaxonomyIn` schema validates them on the write side.
- Forward-only migration `c4f1a2b3d6e7` adds the columns and backfills pre-v2 rows: legacy topic tag → section (default `Customer Experience`), `format` → `Deep Dive`, and the legacy `Industry Move` tag relocated out of `tags` into `story_type`.
- `backend/tests/test_taxonomy.py` — 20 tests: vocab integrity, helper + schema validation, model rejection, all-four-labels save, and backfill SQL semantics.

### Manual tests for the user to run
- `docker compose run --rm backend alembic current` → expect `c4f1a2b3d6e7 (head)`.
- `docker compose run --rm backend pytest` → expect `123 passed`.
- In a Python shell / test: `Post(section="Nonsense")` raises `ValueError`; `Post(section="Customer Experience", format="Brief", story_type="Vendor Launch", tags=["Voice AI"])` saves.

### Architectural changes
- **Data model:** added `section`, `format`, `story_type` (nullable TEXT) to `posts`. Additive, forward-only migration — aligns with PLANNING.md's stated Phase 1 direction. **Fold into PLANNING.md's `posts` table + flip the Phase-1 forward-note when merging the sandbox to main.**
- **Vocabulary lives in code** (`backend/taxonomy.py`), not DB enums — deliberate, per the Data-model decision in `notes/v2-ideas.md`. Consider a `PLANNING-decisions.md` entry capturing "plain-text + code vocab over enums" at review time.
- No API-contract surprises beyond the three additive optional read fields; no scheduler, sourcing, or generation changes.

### Scope notes
- Backfill of `Industry-Move-only` posts falls back to `section = Customer Experience` (no better signal). Documented heuristic; one-line change in the migration if you'd prefer a different default.
- Tag-vocabulary migration (legacy 7 tags → new nested fine tags) and pipeline population of these fields are deliberately deferred to Phase 2, as planned.
